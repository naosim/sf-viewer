# Git 運用ガイドライン（プロジェクト向け）

**⚠️ 重要: git push は自動で実行しない。ユーザーが明示的に要求した場合のみ実施する。**

## コミットメッセージの形式

```
<タイトル>（50文字以内が理想）

<本文>（必要に応じて詳細を記述）
- 変更内容1
- 変更内容2
```

### 例
```
Refactor: SfClientクラス化とIFileSaverによるI/O分離

- execPromise.tsにSfClientクラスを導入（alias/outputDirを保持）
- retrieveSalesforce.tsからfs依存を排除、IFileSaverインターフェースを使用
- process.stdoutをconsole.logに置き換え（process依存の排除）
- index.tsで依存関係を注入する構造に変更
```

## コミット対象の選択基準

### コミットすべきファイル
- プロジェクトのソースコード（`src/` 下）
- プロジェクト設定ファイル（`.gitignore`, `tsconfig.json` など）
- ドキュメント（`README.md` など、プロジェクトに関するもの）

### コミットしないファイル
- 個人のメモ（`memo.md` など）
- 一時ファイル、ログファイル
- 環境依存ファイル（`.env`、ローカル設定）
- `sfdx-project.json`（ローカルで必要だがコミット対象外）
- `output/`（生成された出力）

## ワークフロー

1. **変更作業** → 型チェック (`npx tsc --noEmit`)
2. **ステージング** → `git add <ファイル>`（論理的に関連する変更をまとめる）
3. **コミット** → `git commit -m "タイトル" -m "本文"`
4. **プッシュ** → **行わない**（ユーザーが明示的に要求した場合のみ実施）

## 注意事項

- **git push は決して自動で実行しない。ユーザーからの明示的な要求がある場合のみ実施する。**
- 大きな変更は複数コミットに分割する（例：クラス化、依存排除、リファクタを別々に）
- コミット前に `git diff --staged` で内容を最終確認
- 新規ファイルをコミットするかどうかは、プロジェクトに必要かどうかで判断する
- 不要なファイルは `.gitignore` に追加して追跡対象外とする
